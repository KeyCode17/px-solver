# 0012. `_pxhd` parser and partial synthesis

- **Date:** 2026-05-16
- **Status:** Accepted
- **Deciders:** KeyCode17
- **Related:** ADR-0003, ADR-0009, [`docs/phase/01-foundation.md`](../phase/01-foundation.md), `px-observer` / `px-mimic` (proposed)

## Context

Two real captures (`pedidosya.com.ar`, `hannaandersson.com`) confirm that PerimeterX deployments emit a `_pxhd` cookie whose structure is **stable across sites**:

```
_pxhd = <64-char lowercase hex> ":" <UUIDv1>
       ‚îî‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ server salted ‚îÄ‚îÄ‚îò ‚îî‚îÄ client-side visitor id ‚îÄ‚îò
```

The first half is **server-issued**, derived from at least `(visitor_id, client_ip, server_salt)`, and signed by PX. Without the salt we cannot reproduce it.

The second half is a **UUIDv1** ‚Äî a structured value (`<time-low>-<time-mid>-<time-hi-and-version>-<clock-seq>-<node>`) that is fully synthesizable on the client side and is the same shape browsers naturally generate via `crypto.randomUUID`-adjacent paths inside init.js. It also matches the format of `_pxUuid`, suggesting both fields share a generator inside PX's client code.

`hannaandersson.com` sets `_pxhd` on the **very first response**, before any JS executes, with a server-issued value. `pedidosya.com.ar` only sets it after the challenge round-trip. This means our solver may encounter `_pxhd` in two states:

| State | Source | Trust |
|---|---|---|
| **A. Server-issued** | `Set-Cookie` from the target before init.js runs | Treat as opaque; preserve byte-exactly. |
| **B. Post-challenge** | Refreshed after `/init.js` runs and the sensor POST returns | Same ‚Äî opaque. |
| **C. Client-synthesized** | Generated by us before any request | Only the UUIDv1 half; the hex half is empty or zeroed. |

## Decision

Implement a dedicated `_pxhd` parser and partial synthesizer inside `px-observer` / `px-mimic`:

1. **Parser** (`px_observer::pxhd::parse`) returns `PxHd { hex: [u8; 32], vid: Uuid }` and rejects anything that does not match `^[0-9a-f]{64}:[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$`.
2. **Pure parse + preserve** is the default behavior. The solver never alters a server-issued `_pxhd` it has been handed.
3. **Synthesis** (`px_mimic::pxhd::fresh_uuid_v1`) only emits the UUIDv1 half. The hex half is left empty (`""`) on outbound requests. This matches what a brand-new browser sends before it has seen PX. PX is expected to respond with a real `Set-Cookie: _pxhd=...:...` on the next response, which the solver then captures and preserves.
4. **Never forge the hex half.** Forging it would require the server salt and is out of scope (see ADR-0010). Any attempt would fail signature verification and harm scoring.
5. **Per-site policy.** The `px-observer` records, for each allowlisted domain, whether the site sets `_pxhd` pre-challenge (Hanna) or only post-challenge (PedidosYa). The harvester reads this policy on solve.

## Alternatives considered

- **Treat `_pxhd` as opaque only, no synthesis.** Safer but loses the pre-seeding benefit on sites where PX scores returning-visitor traffic less harshly.
- **Full synthesis (hex + uuid).** Rejected ‚Äî guaranteed signature mismatch; degrades canary success.
- **Replay-only.** Capture a known-good `_pxhd` and replay it. Works briefly but binds the cookie to the original IP and visitor cluster; brittle for production. Useful as a research tool, not a default.

## Consequences

- **Positive:** detector + observer can read `_pxhd` reliably; harvester has a clear policy for what to send when there is no prior cookie state; the synthesis surface is honest about its limits (no fake server signatures).
- **Negative:** no throughput win from synthesis alone ‚Äî `_pxhd` still requires a server round-trip to become valid. Real win comes from combining the UUIDv1 half with cached `_px3` (see ADR-0008).
- **Follow-ups:**
  - `px-observer` records two example captures (pedidosya, hannaandersson) as fixtures.
  - `px-mimic` exposes `fresh_uuid_v1` and `synthesize_pxhd_seed`; both are documented as "client-side only ‚Äî server completes the hex half."
  - Open the next free ADR slot (ADR-0016+) for `_pxvid` synthesis (similar shape, different role) once we have a third PX capture. (ADR-0013 was claimed by the RE methodology ‚Äî see [ADR-0013](0013-re-methodology-and-scope.md).)
